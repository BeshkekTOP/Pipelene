## 1. СТРУКТУРНАЯ СХЕМА САЙТА

### Описание основных блоков и их связей

Структурная схема отображает архитектуру веб-приложения и взаимодействие основных компонентов системы.

### Основные блоки:

#### 1.1. ПРЕЗЕНТАЦИОННЫЙ СЛОЙ (Frontend)
- **Блок навигации** - главное меню, меню ролей (гость, покупатель, менеджер, администратор)
- **Блок аутентификации** - вход, регистрация, выход
- **Блок каталога** - просмотр книг, фильтрация, поиск, детали книги
- **Блок корзины** - добавление/удаление товаров, изменение количества
- **Блок оформления заказа** - форма доставки, подтверждение заказа
- **Блок профиля** - просмотр и редактирование данных пользователя
- **Блок заказов** - история заказов, детали заказа, отмена заказа
- **Блок отзывов** - просмотр, добавление, редактирование, удаление отзывов
- **Блок админ-панели** - управление пользователями, книгами, остатками, отчетами, логами, резервными копиями
- **Блок менеджера** - управление заказами, статистика
- **Блок сообщений** - уведомления об операциях (с возможностью закрытия)

#### 1.2. БИЗНЕС-ЛОГИКА (Backend - Views)
- **Блок views.py** - основные представления (каталог, корзина, профиль)
- **Блок buyer_views.py** - представления для покупателей (отзывы, заказы, профиль)
- **Блок admin_views.py** - представления для администраторов (управление пользователями, книгами, отчетами)
- **Блок manager_views.py** - представления для менеджеров (заказы, статистика)
- **Блок backup_views.py** - представления для резервного копирования
- **Блок sales_views.py** - представления для статистики продаж

#### 1.3. СЛОЙ ДОСТУПА К ДАННЫМ (Models)
- **Блок catalog** - модели Book, Author, Category, Inventory
- **Блок orders** - модели Order, OrderItem, Cart, CartItem
- **Блок users** - модель Profile
- **Блок reviews** - модель Review
- **Блок core** - модель AuditLog (логирование действий)
- **Блок analytics** - модели SalesStats, CustomerStats, TopSellingBook

#### 1.4. СЛОЙ БЕЗОПАСНОСТИ И КОНТРОЛЯ ДОСТУПА
- **Блок decorators.py** - декораторы проверки ролей (guest_required, buyer_required, manager_required, admin_required)
- **Блок middleware.py** - промежуточное ПО для логирования
- **Блок roles.py** - определение ролей и прав доступа

#### 1.5. БАЗА ДАННЫХ
- **PostgreSQL** - основная база данных (в продакшене)
- **SQLite** - база данных для разработки
- **Redis** - кэширование и очереди задач

#### 1.6. ВНЕШНИЕ СЕРВИСЫ
- **Nginx** - веб-сервер, прокси, раздача статических файлов
- **Celery** - асинхронные задачи
- **Docker** - контейнеризация приложения

### Связи между блоками:

1. **Презентационный слой ↔ Бизнес-логика**: HTTP-запросы через URL-маршруты (urls.py)
2. **Бизнес-логика ↔ Слой доступа к данным**: ORM Django для работы с моделями
3. **Слой безопасности ↔ Бизнес-логика**: декораторы проверяют доступ перед выполнением views
4. **Бизнес-логика ↔ База данных**: через Django ORM (PostgreSQL/SQLite)
5. **Бизнес-логика ↔ Redis**: для кэширования и очередей Celery
6. **Nginx ↔ Презентационный слой**: проксирование запросов к Django
7. **Docker ↔ Все компоненты**: контейнеризация всех сервисов

### ДЕТАЛЬНАЯ СТРУКТУРНАЯ СХЕМА (в формате примера)

Схема состоит из трех верхнеуровневых блоков: **Backend**, **Web**, **Database**. Блок **Web** является центральным и связывает **Backend** и **Database**.

#### Верхнеуровневые блоки:

**Backend** (левый верхний угол)
- Связи: стрелка от **Web** к **Backend** (веб-интерфейс взаимодействует с бэкендом)
- Вертикальная линия вниз к компонентам бэкенда

**Web** (центральный верхний блок)
- Связи: 
  - стрелка к **Backend** (слева)
  - стрелка к **Database** (справа)
  - стрелка вниз к блоку **Pages**

**Database** (правый верхний угол)
- Связи:
  - стрелка от **Web** к **Database**
  - стрелка вниз к блоку **DataSet**

#### Компоненты Backend (левый столбец, связанные с блоком "Backend"):

Вертикальная линия от **Backend** вниз, от которой в стороны расходятся стрелки к файлам:

1. `views.py` (web/views.py) - основные представления
2. `buyer_views.py` - представления для покупателей
3. `admin_views.py` - представления для администраторов
4. `manager_views.py` - представления для менеджеров
5. `backup_views.py` - представления для резервного копирования
6. `sales_views.py` - представления для статистики продаж
7. `urls.py` - URL-маршруты
8. `models.py` (catalog/models.py) - модели каталога
9. `models.py` (orders/models.py) - модели заказов
10. `models.py` (users/models.py) - модели пользователей
11. `models.py` (reviews/models.py) - модели отзывов
12. `models.py` (core/models.py) - модели ядра (AuditLog)
13. `models.py` (analytics/models.py) - модели аналитики
14. `serializers.py` - сериализаторы для API
15. `decorators.py` - декораторы проверки ролей
16. `middleware.py` - промежуточное ПО
17. `roles.py` - определение ролей и прав
18. `tests.py` - тесты

#### Компоненты Web → Pages (центральный столбец):

**Pages** (блок под **Web**)
- Связи: вертикальная линия от **Pages** вниз, от которой в стороны расходятся стрелки к HTML-шаблонам

**Пользовательские HTML-шаблоны (правая ветвь от "Pages"):**

2. `buyer/order_detail.html` - детали заказа

#### Компоненты Database:

**DataSet** (блок под **Database**)
- Представляет базу данных (PostgreSQL/SQLite/Redis)

#### Административные HTML-шаблоны (крайний правый столбец, связанные с `admin/dashboard.html`):

Стрелки между административными страницами показывают навигационные связи:

**Связанные с `admin/dashboard.html` (вниз):**
1. `admin/users_list.html` - список пользователей
   - **Связи:** стрелка от `admin/users_list.html` к другим административным страницам

### Общая логика связей:

- **Вертикальные линии:** Показывают, что блок состоит из перечисленных ниже компонентов (например, **Backend** состоит из `views.py`, `urls.py` и т.д.; **Pages** состоит из `home.html`, `cart.html` и т.д.)
- **Горизонтальные стрелки:** Показывают взаимодействие или зависимость между блоками (например, **Web** использует **Backend**, **Web** использует **Database**, `admin/dashboard.html` ведет к `admin/users_list.html`)
- **Иерархия:** Схема показывает иерархическую структуру, где верхние блоки являются более общими, а нижние — более детализированными компонентами

## 2. ФУНКЦИОНАЛЬНАЯ СХЕМА САЙТА

### Описание действий пользователей разных ролей

Функциональная схема отображает последовательность действий пользователей различных ролей при работе с системой.

### Формат схемы (аналогично примеру):

Схема состоит из **четырех вертикальных колонок**, каждая из которых представляет роль пользователя:

1. **ГОСТЬ (Guest)** - крайняя левая колонка
2. **ПОКУПАТЕЛЬ (Buyer/User)** - вторая слева колонка  
3. **МЕНЕДЖЕР (Manager)** - третья слева колонка
4. **АДМИНИСТРАТОР (Administrator)** - крайняя правая колонка

Каждая колонка содержит прямоугольные блоки с действиями, расположенные вертикально один под другим. Блоки связаны вертикальными стрелками, показывающими последовательность или зависимость действий.

**Детальное описание функциональной схемы с полным списком действий и инструкциями по созданию находится в файле `docs/ФУНКЦИОНАЛЬНАЯ_СХЕМА_ДЕТАЛЬНО.md`**

### 2.1. ДЕЙСТВИЯ ГОСТЯ (неавторизованный пользователь)

**Начало работы:**
1. Открытие главной страницы
2. Просмотр каталога книг
3. Просмотр деталей книги
4. Просмотр отзывов на книги

**Регистрация:**
1. Переход на страницу регистрации
2. Заполнение формы (username, email, first_name, last_name, password)
3. Отправка формы
4. Автоматический вход в систему
5. Перенаправление в каталог

**Вход в систему:**
1. Переход на страницу входа
2. Ввод email и пароля
3. Отправка формы
4. Перенаправление в каталог

### 2.2. ДЕЙСТВИЯ ПОКУПАТЕЛЯ (buyer)

**Работа с каталогом:**
1. Просмотр каталога книг
2. Фильтрация по категориям, цене
3. Поиск книг
4. Просмотр деталей книги
5. Добавление книги в корзину

**Работа с корзиной:**
1. Просмотр корзины
2. Изменение количества товаров
3. Удаление товаров из корзины
4. Переход к оформлению заказа

**Оформление заказа:**
1. Заполнение формы доставки (город, адрес, индекс)
2. Добавление примечаний
3. Просмотр итоговой суммы
4. Подтверждение заказа
5. Получение подтверждения об успешном оформлении

**Управление заказами:**
1. Просмотр истории заказов
2. Просмотр деталей заказа
3. Отмена заказа (если статус "обрабатывается")
4. Просмотр статуса заказа

**Работа с отзывами:**
1. Просмотр отзывов на книгу
2. Добавление отзыва с рейтингом (1-5 звезд)
3. Редактирование своего отзыва
4. Удаление своего отзыва

**Управление профилем:**
1. Просмотр профиля
2. Редактирование профиля (имя, фамилия, телефон, адрес)
3. Просмотр истории заказов
4. Просмотр логов активности (только для админа)

### 2.3. ДЕЙСТВИЯ МЕНЕДЖЕРА (manager)

**Все действия покупателя + дополнительно:**

**Управление заказами:**
1. Просмотр всех заказов
2. Просмотр деталей заказа
3. Изменение статуса заказа (обрабатывается → отправлен → доставлен)
4. Просмотр информации о покупателе

**Статистика:**
1. Просмотр панели менеджера
2. Просмотр статистики заказов
3. Просмотр графиков:
   - Динамика заказов по сумме (последние 30 дней)
   - Динамика заказов по количеству (последние 30 дней)
4. Анализ выручки

### 2.4. ДЕЙСТВИЯ АДМИНИСТРАТОРА (admin)

**Все действия менеджера + дополнительно:**

**Управление пользователями:**
1. Просмотр списка пользователей
2. Создание нового пользователя
3. Просмотр деталей пользователя
4. Изменение роли пользователя
5. Просмотр логов активности пользователя
6. Удаление пользователя

**Управление каталогом:**
1. Просмотр списка книг
2. Добавление новой книги
3. Редактирование книги
4. Удаление книги
5. Управление авторами
6. Управление категориями

**Управление остатками:**

### 3.2. ПОСЛЕДОВАТЕЛЬНОСТЬ ДЕЙСТВИЙ ПОКУПАТЕЛЯ

**Сценарий 1: Покупка книги**

1. **Вход на сайт**
   - Открытие главной страницы
   - Нажатие "Вход"
   - Ввод email и пароля
   - Нажатие "Войти"
   - Перенаправление в каталог

2. **Выбор книги**
   - Просмотр каталога
   - Применение фильтров (категория, цена)
   - Нажатие на книгу для просмотра деталей
   - Просмотр описания, цены, наличия, отзывов

3. **Добавление в корзину**
   - Выбор количества
   - Нажатие "Добавить в корзину"
   - Появление сообщения об успехе (можно закрыть крестиком)

4. **Оформление заказа**
   - Переход в корзину
   - Проверка товаров и сумм
   - Нажатие "Оформить заказ"
   - Заполнение формы доставки (город, адрес, индекс, примечания)
   - Просмотр блока "Ваш заказ" с итоговой суммой
   - Нажатие "Оформить заказ на X ₽"
   - Получение подтверждения
   - Перенаправление на страницу успеха

5. **Отслеживание заказа**
   - Переход в "Заказы"
   - Просмотр списка заказов
   - Просмотр деталей заказа
   - Отмена заказа (если статус "обрабатывается")

**Сценарий 2: Добавление отзыва**

1. Просмотр деталей книги
2. Прокрутка к разделу "Отзывы"
3. Нажатие "Добавить отзыв"
4. Выбор рейтинга (1-5 звезд) - при наведении подсвечиваются звезды
5. Ввод текста отзыва
6. Нажатие "Сохранить"
7. Отзыв появляется в списке

### 3.3. ПОСЛЕДОВАТЕЛЬНОСТЬ ДЕЙСТВИЙ МЕНЕДЖЕРА

7. Просмотр статистики в "Менеджер" → "Статистика"
8. Анализ графиков динамики заказов

### Основной поток Pipeline:

```
Разработчики (Django + Python)
    ↓
Git Repository (Version Control)
    ↓
Code Review (Pull Requests)
    ↓
CI Trigger (Git Push/Merge)
    ↓
Тестирование (Unit тесты, Миграции БД, Code Quality)
    ↓
Сборка (Docker Build, PostgreSQL, Static Files)
    ↓
Безопасность (SAST/DAST, Dependency Scan, DB Config Check)
    ↓
Артефакты (Docker Images, Migration Files, Test Reports)
    ↓
Ручное подтверждение (Deploy to Production)
    ↓
Staging Environment (Auto Deploy, Integration Tests, DB Migration Test)
    ↓
Production Environment (Blue-Green Deploy, PostgreSQL Cluster, Load Balancer)
    ↓
PostgreSQL Database (Auto Migrations, Backup/Restore, Monitoring)
    ↓
Мониторинг (Application Logs, Performance Metrics, Error Tracking)
```

### Компоненты инфраструктуры:

1. **Docker**: Контейнеризация
2. **Docker Compose**: Оркестрация контейнеров
3. **PostgreSQL**: База данных
4. **Redis**: Кэш и очереди
5. **Nginx**: Веб-сервер и прокси
6. **Celery**: Асинхронные задачи

### Процесс развертывания:

1. **Разработка**: Изменения в коде
2. **Коммит**: git commit
3. **Сборка**: docker-compose build
4. **Тестирование**: pytest (опционально)
5. **Развертывание**: docker-compose up -d
6. **Миграции**: Автоматически при старте
7. **Проверка**: Логи и health checks
